Running & Understanding the Ad Click Aggregation Backend

This repo contains a working backend that counts ad clicks two ways at once: fast and approximately for dashboards, and slowly but exactly for billing. This guide explains every concept it uses, gets it running on your machine, and walks a real click through the whole pipeline.

01What is this project?

There are two things in this repository:

  • index.html — a system-design document. It's the "architecture on paper": requirements, capacity math, API design, trade-offs. Open it in a browser and read it like a blog post.
  • The implementation — everything under packages/, services/, and infra/. It's that design actually built, in TypeScript, runnable entirely on your laptop with zero AWS account and zero cloud cost.

The problem being solved, in one sentence: when a user clicks an ad, the advertiser gets billed for it — so every click must be counted exactly once, even though networks retry, clients resend, and bots exist.

The design docs that drove each piece live in docs/superpowers/specs/ (what to build & why) and docs/superpowers/plans/ (step-by-step build plans). If you ever wonder "why was it done this way?", the spec usually says.

02The big picture

The core architectural idea — worth internalizing before anything else — is that the system runs two parallel lanes off the same stream of click events:

                          ┌─────────────────────────────────────────────────────┐
                FAST LANE │  aggregator: dedup → 1-min windows → add to counts  │  seconds old,
                (approx.) │  (allowed to be slightly wrong)                     │  for dashboards
                          └────────▲──────────────────────┬─────────────────────┘
 user clicks ad                    │                      ▼
      │                            │               ┌──────────────┐      ┌───────────┐
      ▼                            │               │   DynamoDB   │◀─────│ query-api │◀── dashboard
┌────────────────┐        ┌────────┴────────┐      │  aggregates  │      └───────────┘
│ click-redirect │───────▶│  Kinesis stream │      └──────────────┘
│ (302 first,    │        │  ad-clicks-raw  │
│  count later)  │        └────────┬────────┘
└────────────────┘                 │
                          ┌────────▼────────────────────────────────────────────┐
               EXACT LANE │  archiver: EVERY raw event → S3 as Parquet files    │  hours old,
               (billing)  │  (duplicates included — on purpose!)                │  money-grade
                          └────────┬────────────────────────────────────────────┘
                                   ▼
                    reconcile: DuckDB COUNT(DISTINCT cid) over S3
                                   ▼
                       statements table ──▶ billing-api ──▶ invoice
Why two lanes?

Dashboards need a number in seconds and can tolerate it being slightly off. Invoices need a number that is exactly right and can wait until tomorrow. No single system does both well, so this one doesn't pretend to — the fast number is explicitly labeled "exact": false and the billing number "exact": true. Almost every design decision in the codebase serves one lane or the other.

The second key idea: delivery is "at-least-once", so counting must be deduplicated. Networks time out and retry. The same click event can arrive two or three times. Each lane handles this differently:

  • The fast lane dedups at ingest: each click has a unique id (cid), and a Redis+DynamoDB check drops a cid it has seen before.
  • The exact lane doesn't filter at all — it archives everything, duplicates included, and dedups at query time with SQL's COUNT(DISTINCT cid). That makes reprocessing safe: replaying the same data twice gives the same answer.

03Concepts primer — the things you're "assumed to know," explained

Monorepo & pnpm workspaces

A monorepo is one git repository containing many small packages instead of one big app. This repo uses pnpm "workspaces" to manage them. The file pnpm-workspace.yaml declares three folders of packages:

  • packages/* — libraries: reusable logic, imported by other packages. They don't "run".
  • services/* — runnable processes: HTTP servers and stream consumers. Each has a start script.
  • infra/* — setup scripts: create the local AWS resources (stream, tables, bucket).

Two pnpm commands do 90% of the work:

pnpm -r <script>                  # run <script> in EVERY package ("-r" = recursive), e.g. pnpm -r test
pnpm --filter @app/db <script>    # run <script> in ONE package, by its name in package.json

Packages depend on each other with "@app/db": "workspace:*" in their package.json — that means "link the local folder, not something from the internet."

TypeScript, ESM, and the dist/ folder

Everything is TypeScript compiled by tsc into a dist/ folder per package. When services/query-api imports @app/db, Node actually loads packages/db/dist/index.js — so if you haven't built a package, imports of it fail. That's why the setup below includes pnpm -r build. (Imports end in .js even in .ts files — that's an ESM convention, not a typo.)

Docker, Compose, and colima

Docker runs software in isolated containers — here, the databases. Docker Compose starts a whole set of them from one file (docker-compose.yml): Postgres, Redis, and LocalStack. On macOS without Docker Desktop, colima provides the Linux VM Docker needs — you just run colima start once per boot.

LocalStack — "fake AWS on your laptop"

The design targets real AWS services, but you don't need an AWS account. LocalStack is a container that emulates AWS APIs on http://localhost:4566. The code uses the real AWS SDK — the only difference is one environment variable, AWS_ENDPOINT_URL, pointing at LocalStack. Unset it (and supply real credentials) and the identical code talks to real AWS. There is deliberately no if (isLocal) branching anywhere.

The storage cast — what each database does here

TechnologyThink of it as…Used here for
Kinesis (via LocalStack)A conveyor belt of events. Producers put records on; multiple independent consumers read them in order. Split into parallel lanes called shards.The ad-clicks-raw stream every click flows through. Locally it has 2 shards.
DynamoDB (via LocalStack)A giant, fast key-value dictionary. No SQL joins; you fetch by key.Three tables: click-aggregates (live counts), click-dedup (seen click-ids), click-statements (billing output).
S3 (via LocalStack)An infinite file drive in the cloud.The raw archive: every click event, stored forever as Parquet files under s3://ad-clicks-raw/dt=YYYY-MM-DD/.
RedisAn in-memory dictionary measured in microseconds. Loses data on restart — that's accepted.The dedup fast-path: "have I seen click-id X in the last 10 minutes?"
Postgres + PrismaA classic relational database; Prisma is the TypeScript library that talks to it with generated types.The "control plane": advertisers, campaigns, ads, API keys — the slow-changing business data.
DuckDBSQLite's analytics cousin: a SQL engine embedded inside the Node process — no server. Reads/writes Parquet directly on S3.The nightly reconciliation query (the local stand-in for AWS Athena).

Smaller supporting cast

  • Parquet — a compressed, columnar file format; the standard for analytics data at rest.
  • Fastify — the web framework the HTTP services use (like Express, but faster and typed).
  • Zod — runtime validation: ClickEventSchema.safeParse(input) rejects malformed events at the door.
  • Vitest — the test runner. Unit tests are pure; integration tests talk to the real local containers.
  • HMAC signature (sig) — every click URL carries a cryptographic signature computed from its fields with a per-advertiser secret. A forged or tampered URL fails verification. This is the system's authentication for clicks.

05Getting it running

Prerequisites (one-time)

ToolCheckInstall (macOS)
Node.js ≥ 24node -vbrew install node (or nvm)
pnpm ≥ 9pnpm -vnpm install -g pnpm
Docker CLI + colimadocker psbrew install colima docker docker-compose
Two macOS gotchas (both hit during development)
  • If docker compose (with a space) says "not a docker command", link the plugin: mkdir -p ~/.docker/cli-plugins && ln -sfn /opt/homebrew/opt/docker-compose/bin/docker-compose ~/.docker/cli-plugins/docker-compose
  • If you have Homebrew's Postgres installed, it silently occupies port 5432 and shadows the Docker one — you'll get baffling auth failures. Fix: brew services stop postgresql@17 (or whatever version brew services list shows).

The bootstrap sequence

  1. Start the Docker VM (once per boot): 
    colima start
  2. Start the databases (LocalStack + Postgres + Redis) and wait for all three to report healthy: 
    cp .env.example .env        # first time only
docker compose up -d
docker compose ps           # repeat until all 3 say "healthy" (~30s)
  3. Install & build: 
    pnpm install                # also auto-generates the Prisma client (postinstall)
pnpm -r build               # compiles every package's dist/ — required before running services
  4. Create the AWS resources in LocalStack (idempotent — safe to re-run): 
    pnpm --filter @app/infra-localstack bootstrap
# → "LocalStack resources ready: Kinesis stream, dedup table, hot aggregate table,
#    statements table, raw archive bucket."
  5. Create the database schema, then seed demo data: 
    DATABASE_URL=postgresql://app:app@localhost:5432/app \
  pnpm --filter @app/db exec prisma migrate deploy

DATABASE_URL=postgresql://app:app@localhost:5432/app \
  pnpm --filter @app/db seed
# → "Seed complete. Demo API key (save this, it is not stored raw): 3f9c…"
    Save that API key!

    The seed prints a demo API key — copy it somewhere. It's your Bearer token for the query and billing APIs, and it's stored only as a hash (it cannot be recovered — though re-running the seed mints a fresh one).

  6. Prove everything works — run the full test suite (85 tests, unit + integration against the live containers): 
    DATABASE_URL=postgresql://app:app@localhost:5432/app pnpm -r test
Why is DATABASE_URL typed inline every time?

The Prisma CLI auto-loads .env, but the Prisma client library (used at runtime and in tests) reads only the real process environment. Prefixing the variable is the simplest reliable way. Alternatively, load the whole .env into your shell once per terminal: set -a; source .env; set +a — then you can drop the prefixes for that session.

06A click's journey — hands-on demo

This is the fun part: fire one real click and watch it move through every store. You'll need a few terminals — each long-running process gets one. First, in every terminal, load the environment:

set -a; source .env; set +a

Step 1 — start the processes

TerminalCommandWhat it is
T1KINESIS_SHARD_ID=shardId-000000000000 pnpm --filter @app/aggregator startFast-lane consumer, shard 0
T2KINESIS_SHARD_ID=shardId-000000000001 pnpm --filter @app/aggregator startFast-lane consumer, shard 1
T3KINESIS_SHARD_ID=shardId-000000000000 pnpm --filter @app/archiver startExact-lane archiver, shard 0
T4KINESIS_SHARD_ID=shardId-000000000001 pnpm --filter @app/archiver startExact-lane archiver, shard 1
T5pnpm --filter @app/click-redirect startThe redirect endpoint (:3000)
T6pnpm --filter @app/query-api startDashboard read API (:3002)
T7OPS_TOKEN=dev-ops-token pnpm --filter @app/billing-api startBilling API (:3003)
Why one process per shard?

Kinesis parallelism = shards. Each consumer process owns exactly one shard — that's the design's scaling model (in production, ECS runs one container per shard automatically; locally, you are the orchestrator). Clicks are spread across shards pseudo-randomly, so with only shard-0 consumers running, roughly half your clicks would be invisible. Running both shards makes the demo deterministic. Skip T2/T4 if you don't mind re-firing clicks until one lands on shard 0.

Step 2 — forge a valid, signed click URL

A real click URL is generated by the ad server at render time, HMAC-signed with the advertiser's secret so it can't be forged or tampered with. We'll play ad server, using the seeded advertiser's secret (seed-signing-secret) and the seeded ad:

node --input-type=module -e '
import { createHmac } from "node:crypto";
const f = {
  cid: "clk_demo_" + Date.now(),          // unique click id — the dedup key
  ad_id: "seed-ad-1",
  campaign_id: "seed-campaign-1",
  pub_id: "seed-publisher-1",
  ts: new Date().toISOString(),
};
const sig = createHmac("sha256", "seed-signing-secret")
  .update([f.cid, f.ad_id, f.campaign_id, f.pub_id, f.ts].join("|"))
  .digest("hex");
const r = encodeURIComponent("https://advertiser.example.com/landing");
console.log("http://localhost:3000/click?" + new URLSearchParams({ ...f, sig, r }));
'

Step 3 — fire it, and watch each layer react

# Fire the click (the URL from step 2). -i shows headers:
curl -i "<PASTE THE URL>"
# → HTTP/1.1 302 Found,  location: https://advertiser.example.com/landing
#   The user is already gone — everything below happened AFTER the redirect.

# (instant)  The dedup ledger in Redis now knows your cid:
docker compose exec redis redis-cli keys 'click:clk_demo_*'

# (within ~35s)  The raw event is archived to S3 as Parquet:
docker compose exec localstack awslocal s3 ls s3://ad-clicks-raw/ --recursive

# (after 2–3 min — see the note below)  The count lands in DynamoDB:
docker compose exec localstack awslocal dynamodb scan \
  --table-name click-aggregates --max-items 5
Why does the live count take 2–3 minutes?

Deliberate design, not slowness: the aggregator counts clicks into 1-minute windows by event time, then holds each window open 2 extra minutes (the "watermark") for stragglers that arrive late, then flushes. So a click becomes visible in the aggregate 2–3 minutes after its timestamp. Until then the query below honestly answers clicks: 0 for the current window.

# Query the dashboard number (use the API key the seed printed):
curl -s -H "Authorization: Bearer <YOUR_SEED_API_KEY>" \
  http://localhost:3002/v1/ads/seed-ad-1/aggregates
# → {"adId":"seed-ad-1","windowStart":"…","clicks":1,"exact":false,"asOf":"…"}

Step 4 — try to cheat (and fail)

# Replay the EXACT same URL — same cid — five more times:
for i in 1 2 3 4 5; do curl -s -o /dev/null "<SAME URL>"; done
# The user still gets redirected every time (that part must never break),
# but the count does NOT go up: Redis SET NX says "seen it", duplicate dropped.

# Tamper with one signed field (change ad_id in the URL) → 400 invalid_request.
# Point r at a different site → 400 (open-redirect protection: r must equal the
# landing URL stored in Postgres for that ad). All rejections return the SAME
# error body, so an attacker can't probe which ad_ids exist.

Step 5 — run billing, like the nightly job would

# Recompute today's exact statement from the raw S3 archive (ops-only endpoint):
curl -s -X POST -H "X-Ops-Token: dev-ops-token" \
  http://localhost:3003/v1/reconciliation/$(date -u +%F)/rerun
# → {"date":"2026-…","campaignsReconciled":1}

# Read the billing-grade statement (note "exact": true — and note that even if
# you fired duplicates in step 4, billedClicks counts each cid ONCE, because
# reconciliation is COUNT(DISTINCT cid) over the unfiltered archive):
curl -s -H "Authorization: Bearer <YOUR_SEED_API_KEY>" \
  http://localhost:3003/v1/campaigns/seed-campaign-1/statement
# → {"campaignId":"seed-campaign-1","period":"…","billedClicks":1,
#    "excludedInvalidClicks":0,"exact":true,…}

That's the whole system: one tap became one row on an invoice — not zero, not two — with a fast approximate number along the way.

07How each piece works (and the "why" behind it)

The hot path: services/click-redirect

The only user-facing latency in the system. Order of operations is sacred:

  1. Validate the query shape (Zod), look the ad up in an in-memory cache (refreshed from Postgres every 30s — no DB call on the hot path), verify the HMAC sig, and check r equals the stored landing URL (open-redirect protection — never redirect to a client-supplied URL).
  2. Send the 302. The user's browser is now leaving.
  3. Only then, publish the event to Kinesis via setImmediate — fire-and-forget, so a slow or failing Kinesis can never delay a human.

Every rejection returns the identical 400 {"error":"invalid_request"} — deliberately indistinguishable, so callers can't enumerate valid ad ids by comparing error messages.

The idempotency key: why cid is generated client-side

The one decision everything depends on

The click id is minted before the first thing that can fail. If a retry resends the same cid, downstream dedup can recognize it. If the server minted a fresh id per attempt, a retry would be indistinguishable from a new click and no amount of downstream cleverness could ever fix it.

Exactly-once counting: packages/click-dedup

Two tiers, checked in order:

  • Redis fast path: SET click:<cid> 1 NX EX 600 — one atomic operation meaning "claim this id if unclaimed, expiring in 10 minutes." Already claimed → duplicate → drop.
  • DynamoDB backstop: only consulted when Redis says "new" — a conditional write (attribute_not_exists(cid)). If Redis restarted and forgot, DynamoDB still remembers. This is what makes it a genuine backstop rather than a redundant second check.

Windowing: packages/windowed-aggregator

Counts are bucketed per (ad, minute) using the click's event time (when the user tapped), not arrival time. A window stays open 2 minutes past its end (the watermark) to admit stragglers, then flushes. Two subtleties the code fought hard for:

  • Flush uses DynamoDB ADD (atomic increment), never overwrite — so a late flush for an already-flushed window adds to the count instead of clobbering it.
  • Retry-safety: a window's in-memory count is only removed after its flush succeeds, per-ad (commitFlushed). If DynamoDB throttles mid-flush, nothing is double-added on retry and nothing is lost. (The review process caught a compounding double-count bug in an earlier version of this logic — the fix is why the API looks the way it does.)

The exact lane: archiver → DuckDB → statements

  • services/archiver writes every raw event to S3 — no filtering, duplicates included — batched into Parquet files, partitioned by the event's own date (so a retry that straddles midnight still lands with its original in the same partition — another review-caught billing bug).
  • reconcileDate (in packages/parquet-archive) runs one DuckDB SQL query over the day's partition: COUNT(DISTINCT cid) per campaign, with a parameterized exclusion set reserved for confirmed-fraud clicks (empty until sub-project 6 lands).
  • statements-store writes results with a full overwrite — the opposite of the fast lane's ADD — because each reconciliation is a from-scratch recompute. Re-running it must replace the statement, never stack on top.
Why archive the duplicates?

Because it makes reprocessing boring. The archive is an immutable record of what happened; correctness lives in one SQL query. Found a bug? Re-run the query. Fraud model updated? Re-run the query. Same input → same answer, every time. If the archive were pre-filtered, every replay would depend on the filter having been right forever.

The read APIs: tenant isolation

Both read APIs resolve your Bearer key to an advertiser (keys stored as SHA-256 hashes; revoked keys fail closed), then check ownership before touching data. Asking about an ad that doesn't exist and an ad owned by someone else return byte-identical 404s from a single code branch — so a competitor cannot even learn which ad ids exist. The billing API's ops endpoint (/reconciliation/:date/rerun) uses a completely separate credential (X-Ops-Token); advertiser keys don't work there and vice versa.

How the tests are organized

  • Pure unit tests (schema, signature, windowing, batching) — no infrastructure, milliseconds, time passed in explicitly rather than mocked clocks.
  • Integration tests (db, dedup, stores, parquet-archive, infra) — hit the real local Postgres/Redis/LocalStack, which is why the containers must be up.
  • Service tests — Fastify's inject() with dependency-injected fakes: every route's logic is tested without any real network or database. Each service exports a pure buildApp(deps); only the thin server.ts wires real clients, and that file is deliberately untested.

08Deliberate gaps & limits (so they don't surprise you)

  • Fraud scoring (sub-project 6) isn't built yet. The reconciliation query's exclusion seam exists but receives an empty set; excludedInvalidClicks is always 0 for now.
  • Local-first only. No Terraform/CDK, no deployment pipeline. Pointing at real AWS is "just" configuration, but productionizing is out of scope.
  • Consumer resilience is minimal. If Kinesis throttles or an iterator expires, a consumer crashes and restarts from "now" — in-memory windows/buffers are lost. Fine for the approximate lane (the archive + reconciliation recovers billing), tracked as the top hardening item for the exact lane.
  • Fire-and-forget publish. If Kinesis is down past SDK retries, a click is redirected but not counted (logged, not silently ignored). The doc's own trade-off; a durable outbox is the known fix.
  • Windows/aggregates are approximate by contract. Never bill from query-api; that's what statements are for.

10Glossary

TermMeaning here
cidClick ID — the unique, client-minted identifier of one click; the dedup key for everything.
At-least-once deliveryThe guarantee streams actually give you: nothing is lost, but things may arrive twice. The reason dedup exists.
IdempotentSafe to do twice with the same result as once. Seeds, bootstraps, and reconciliation here are all idempotent on purpose.
ShardOne parallel lane of a Kinesis stream. One consumer process per shard.
Partition keyDecides which shard a record goes to. Salted here (ad_id#0-7) so one viral ad can't overload a single shard.
Event time vs. processing timeWhen the click happened vs. when we saw it. Counting uses event time; the gap between them is "lateness".
WatermarkHow long a window waits for late events before flushing (2 minutes here).
Control planeThe slow-changing business data (advertisers, campaigns, ads, keys) in Postgres — as opposed to the high-volume click "data plane".
Hot pathCode in the way of a waiting user (the redirect). Everything else is deliberately kept out of it.
ReconciliationThe from-scratch recompute of exact billing numbers from the raw archive.
Tenant isolationAdvertiser A can never read (or even detect the existence of) advertiser B's data.
DI / buildApp(deps)Dependency injection — routes receive their collaborators as arguments, so tests pass fakes and never need real infrastructure.
