00

Overview

A user sees an ad and clicks it. Somewhere downstream, that single tap has to become exactly one row in one advertiser's bill — not zero, because the network dropped it; not two, because a retry duplicated it; and not one, if it turns out later the "user" was a bot farm. This document designs the service that sits between the click and the invoice.

The reason this is a genuinely hard system rather than a `COUNT(*) GROUP BY ad_id` query is that every input to it is unreliable in a specific, well-understood way: delivery is at-least-once, clocks and networks make events arrive out of order, and a meaningful fraction of traffic isn't a human being at all. The rest of this document is largely about absorbing those three failure modes without either losing a real click or inventing one that never happened.

01

Functional requirements

In scope

  • Accept a click event fired from an ad redirect/tracking link and forward the user to the advertiser's landing page with minimal added latency.
  • Deduplicate retried or redelivered click events so each real click is counted exactly once in any aggregate that money depends on.
  • Aggregate click counts per ad, campaign, and advertiser over rolling real-time windows (minute-level) and fixed billing windows (daily/monthly).
  • Serve fast, approximate real-time aggregates for dashboards and campaign budget-pacing checks.
  • Produce exact, fully reconciled aggregates for advertiser invoicing, traceable back to the raw events that produced them.
  • Detect and label suspicious or fraudulent clicks asynchronously, and exclude confirmed-invalid clicks from billed counts before an invoice is finalized.
  • Support backfilling and reprocessing historical aggregates — from a fixed-logic bug, a fraud-model update, or an advertiser dispute.
  • Expose per-advertiser, per-campaign query APIs, scoped so one advertiser can never read another's click data.

Explicitly out of scope

  • Ad impression tracking. Impressions outnumber clicks by 2–3 orders of magnitude and have a very different NFR profile (loss-tolerant, no fraud-billing stakes); it shares infrastructure patterns with this system but is a separate pipeline.
  • Real-time bidding / ad-serving decisions. This system counts what already happened. It has no say in which ad was shown or which bid won.
  • Conversion tracking & multi-touch attribution. This system hands off click_id and event-time as the attribution anchor; stitching that click to a later purchase is a downstream system's job.
  • Advertiser-facing dashboard UI. This document covers the pipeline and the API it exposes, not the frontend that renders it.
02

Non‑functional requirements

The requirement that shapes almost every decision below: this pipeline has two customers with incompatible needs from the same event stream — a dashboard that wants a number in seconds, and an invoice that must be exactly right. Meeting both means never pretending the fast number and the correct number are the same thing.

03

Capacity estimation

Assumptions

  • Ad network: ~15,000 publisher placements serving ~500,000 active campaigns (~3 ads/campaign on average, ~1.5M active ad_ids).
  • 24 billion ad impressions/day across the network — context for scale, not this system's subject.
  • Blended click-through rate of 0.2% across formats — low, and deliberately not inflated; click volume is genuinely much smaller than impression volume.
  • Diurnal peak-to-average ratio of 3.2×, plus a further 6.5× burst multiplier for campaign launches and coordinated bot storms, which are exactly the moments this system is most stressed and most needed.

04

API design

Three surfaces, in decreasing order of how much they can't go wrong: the click redirect every ad tap flows through, a batch ingestion path for clients that can't use a redirect, and a query API the dashboard and billing system both read from.

Click redirect (hot path)

Ad click tracking is fundamentally a redirect service: the link the user taps points here first, and this service's only user-facing job is to get them to the advertiser's page fast. Everything about counting the click happens off that critical path.

click-handler.js — Fastify, fire-and-forget event emission 
import { verifySignature } from './sig.js';
import { KinesisClient, PutRecordCommand } from '@aws-sdk/client-kinesis';

const kinesis = new KinesisClient({ region: process.env.AWS_REGION });

export default async function routes(app) {
  app.get('/click', async (req, reply) => {
    const q = req.query;
    if (!verifySignature(q)) return reply.code(400).send({ error: 'invalid_signature' });

    // redirect first — nothing below this line may add latency to it
    reply.redirect(302, decodeURIComponent(q.r));

    // best-effort durable enqueue, off the response path (see §6.1 for why cid must
    // originate client-side and §6.6 for why this is a PutRecord, not a blocking await)
    setImmediate(() => {
      kinesis.send(new PutRecordCommand({
        StreamName: 'ad-clicks-raw',
        PartitionKey: `${q.ad_id}#${Math.floor(Math.random() * 8)}`, // salted — see §6.3
        Data: Buffer.from(JSON.stringify({ ...q, receivedAt: Date.now() })),
      })).catch((err) => app.log.error({ err, cid: q.cid }, 'click enqueue failed'));
    });
  });
}

06Deep dives

Seven places where "just count the clicks" turns out to have a non-obvious answer — each one is where a naive implementation is quietly wrong, and the fix is small once you see why.

6.1Exactly‑once counting & idempotency

Nothing in this pipeline is exactly-once by default. A client retries a redirect that timed out; the click service's PutRecord succeeds but the ack is lost, so it retries; a consumer crashes after processing a batch but before checkpointing, so Kinesis redelivers it. Every one of those is a legitimate, expected at-least-once duplicate — the system's job is to make the count exactly-once even though delivery never is.

The one decision everything else depends on

cid is generated once, client-side (or at the edge, deterministically), before the first retryable hop. A retry resends the same ID. If the ID were minted fresh on each attempt, no downstream dedup logic could ever tell a retry apart from a second, real click — the idempotency key has to be older than the first opportunity to fail.

  • Fast-path dedup: SET click:{cid} 1 NX EX {dedupWindowSeconds} in Redis — one atomic call. If the key already existed, this is a duplicate: acknowledge it (so it stops being redelivered) but don't count it.
  • Durable-path backstop: a DynamoDB conditional write (attribute_not_exists(cid)) on the billing-critical aggregation path. Redis is fast but a restart can lose its dedup set; DynamoDB is the guard that survives that.
  • Window sizing: long enough to cover realistic redelivery lag (SQS/Kinesis retry timing, consumer crash-recovery), short enough to bound memory — 10 minutes costs about 1.1 GB at burst (§03), so erring longer is cheap if redelivery windows turn out wider in practice.

6.2Late data & watermarks

A click's event-time (when the user tapped it) and its processing-time (when this pipeline saw it) routinely diverge — sometimes by hours. The clearest case: a mobile SDK buffers clicks while the device is offline and flushes the whole queue on next launch. A 1-minute real-time window keyed on ad_id has to have a policy for when it's allowed to stop waiting and ship a number.

  • Watermark: the real-time layer closes and flushes the window for minute T once stream event-time has advanced to T+2min — a fixed, bounded allowed-lateness, not an unbounded wait for stragglers.
  • Late events aren't dropped, they're deferred: anything arriving after its window has closed still lands untouched in the S3 raw archive. It simply doesn't get to be in the fast, approximate number — it gets to be in the exact one.
  • This is what makes the batch layer non-optional: the reconciliation job runs hours later, over the complete raw archive, with no lateness deadline at all. It's the only layer honest late data can always be correct in.
Concretely

A click at 09:14 from a phone that goes offline and doesn't sync until 15:40 the same day is not an edge case worth special-casing away — it's routine. The real-time dashboard will simply never show it in the 09:14 bucket. The next day's advertiser statement will.

6.3Hot keys & partition skew

Click volume is not evenly spread across 1.5M ad_ids. A single viral or heavily-boosted campaign can account for 5% of a day's total clicks on its own — at the 48M/day baseline that's 2.4M clicks on one ad_id, averaging ~28/s, which is unremarkable. During a genuine spike (or a bot attack aimed at burning one advertiser's budget), that same key can burst past 5,000 clicks/s — which saturates a single Kinesis shard's 1,000 records/s limit and a single DynamoDB partition many times over, while every other shard in the stream sits idle.

  • Salted partition key: the click service writes to Kinesis with key {ad_id}#{random 0-7}, not the bare ad_id (see the click-handler snippet in §04) — spreading one hot ad's traffic across up to 8 shards instead of pinning it to one.
  • Local pre-aggregation: each consumer sums the records it personally sees for a window in memory and flushes one delta upsert per (ad_id, window) when the window closes — turning thousands of per-click writes into one batched increment per consumer per window. This is the single biggest lever on DynamoDB write amplification, hot key or not.
  • Adaptive counter sharding: only for a key that's still hot after pre-aggregation, write to {ad_id}#{shard} sub-counters in DynamoDB and sum across shards on read — applied selectively to detected hot keys, not universally, since the other 99.9% of ad_ids never need the extra read-side fan-in cost.

6.4Fraud & invalid traffic

This is the actual business problem, not a bolt-on. A meaningful share of raw clicks on any ad network are not a genuine, billable human interest signal: click farms, competitor click fraud aimed at burning a rival's budget, and — overwhelmingly the most common source in practice — link-preview bots (Slack, iMessage, Discord, email clients) firing a GET against the redirect URL just to render a preview card, with no human involved at all.

Synchronous checks (kept deliberately minimal — they sit in front of the redirect)

  • Signature validation (sig, §04) — rejects forged or replayed click URLs outright.
  • Coarse velocity check per IP/device fingerprint — not a verdict, just a flag attached to the event.
  • Known preview-bot detection via header conventions (Purpose: prefetch, known Slackbot/iMessage/Discord user-agent strings) and requiring a follow-up landing-page beacon before a click is eligible to be billable — a redirect alone proves a link was fetched, not that a person looked at it.

Asynchronous scoring (never blocks a count)

A third Kinesis consumer runs fraud scoring off the same stream, writing a verdict per cid to a side table on its own schedule — the deliberate architectural choice is that this consumer's speed affects nothing about how fast a click gets counted, only how fast it gets a fraud label.

SignalVerdictEffect on real-time countEffect on billing statement
Passed all checkslegitimateCountedBilled
Elevated velocity, unresolvedsuspiciousCountedHeld pending review; excluded if unresolved by close
Confirmed bot / farm patterninvalidCounted (unchanged, real-time is never rewritten)Excluded, credited

Trust-and-safety policy — exactly how aggressively to exclude "suspicious but unconfirmed" clicks — is a business decision this pipeline makes available fast, not one it hardcodes. Its job is to get the signal into the reconciliation job before an invoice is final, every time.

6.5Approximate counting: when sketches earn their keep

Count-Min Sketch and HyperLogLog are the reflexive answer to "aggregate at scale" in a lot of system-design writing, and reaching for them here by default would be a mistake. This system's own capacity numbers (§03) show exact counting is cheap at this scale — ~1.5M keys, a write rate that collapses to a few hundred thousand pre-aggregated upserts a minute. A sketch trades accuracy for memory you don't need to save, on a number that's headed for an invoice. Default to exact (DynamoDB atomic ADD / Redis INCR).

Two adjacent questions in this same domain do earn a sketch, because their cardinality is genuinely large and their answer is never billing-critical:

  • Unique-reach dashboards — "how many distinct users clicked this campaign" over a user-ID space in the hundreds of millions. HyperLogLog gets ~0.8% error at a tiny fraction of an exact hash set's memory, which is worth it specifically because exact unique-user tracking at that cardinality is expensive, unlike click counting.
  • Real-time frequency capping in the ad-serving decision itself (a different system than this one) — "has this user seen this ad 3 times already" needs an O(1) approximate check cheap enough to run inline on every ad request; an exact per-user-per-ad counter table would not be.

The discipline is the same one that picks a chart type before picking colors: ask what the number is actually for before reaching for the structure that makes it cheap.

6.6Node.js consumer architecture

Two honest constraints shape this section: Kinesis's consumption model gives at most one active reader per shard per consumer group, and Node's event loop is single-threaded. Neither is a reason to avoid Node.js here — both just mean the scaling lever is shard count, not thread count, and the process model has to fit how Kinesis actually hands out work.

  • Fargate, not Lambda, for real-time aggregation: holding partial window sums in memory across many records needs a long-lived process. Lambda's execution environment can persist state between invocations, but it isn't guaranteed to, which is the wrong foundation for correctness-sensitive windowing. One Fargate task per shard, using @aws-sdk/client-kinesis's SubscribeToShard (enhanced fan-out) for push-based delivery instead of polling.
  • Lambda where it fits: the Firehose→S3 transform and the fraud-scoring consumer are both stateless, one-record-in-one-record-out shapes — a good fit for Lambda's model, unlike windowed aggregation.
  • worker_threads for the CPU-bound slice: HMAC verification and local sketch/hash updates run per-record and can measurably block the event loop under burst; offloading that slice to a worker pool keeps the main loop free to keep pulling records, without pretending Node parallelizes the whole pipeline for you.
  • Horizontal scale tracks shard count directly: ~4 tasks at steady peak, ~20 at the burst design target (§03) — ECS service auto-scaling keyed off Kinesis GetRecords.IteratorAgeMilliseconds, not raw CPU, since a lagging iterator is the real signal this fleet is undersized.
aggregator.js — windowed pre-aggregation, one task per shard 
import { KinesisClient, SubscribeToShardCommand } from '@aws-sdk/client-kinesis';
import { dedupe } from './dedup.js';   // Redis SET NX, §6.1
import { flushWindow } from './store.js'; // DynamoDB batched upsert

const WINDOW_MS = 60_000;
const WATERMARK_MS = 2 * 60_000; // §6.2 allowed lateness
let windows = new Map(); // windowStart -> Map

function bucketFor(eventTimeMs) {
  return Math.floor(eventTimeMs / WINDOW_MS) * WINDOW_MS;
}

async function handleRecord(evt) {
  if (!(await dedupe(evt.cid))) return; // duplicate — already counted, drop

  const bucket = bucketFor(new Date(evt.ts).getTime());
  const w = windows.get(bucket) ?? windows.set(bucket, new Map()).get(bucket);
  w.set(evt.ad_id, (w.get(evt.ad_id) ?? 0) + 1); // local, in-memory, no network call per click
}

setInterval(async () => {
  const closeBefore = Date.now() - WATERMARK_MS;
  for (const [bucket, counts] of windows) {
    if (bucket + WINDOW_MS > closeBefore) continue; // still within allowed lateness
    await flushWindow(bucket, counts); // one batched upsert per ad_id in this window
    windows.delete(bucket);
  }
}, 5_000).unref();

// wired via SubscribeToShardCommand's event stream — one shard, one task, push-based

6.7Backfill & reprocessing

The reconciliation job never patches an existing number — it recomputes the exact sum for a date range from raw events in S3, every time. That's only safe because of two properties established earlier: dedup by cid is idempotent (§6.1) and summation is order-independent, so replaying the same raw partition twice produces the same statement, not a double-counted one. That single property is what makes backfill a safe, boring operation instead of a risky one.

  • Trigger examples: a fraud model ships v2 and needs last quarter re-scored; a windowing bug is found and fixed; an advertiser formally disputes a statement.
  • Mechanism: POST /v1/reconciliation/{date}/rerun (§04) re-runs Glue/Athena over the untouched S3 partitions for that range and overwrites the statements table — the raw archive is never mutated, only re-read.
  • Schema evolution: new fields added to the click event over time are handled by Parquet/Glue schema evolution; a reconciliation query over older partitions simply sees those fields as absent, with defaults applied at query time rather than requiring a rewrite of historical data.

Closing notes

What's deliberately not built here: impression tracking, ad-serving/bidding logic, and attribution beyond the click itself. All three are legitimate systems in their own right — none of them belong in the critical path of turning one tap into one correct, defensible row on an invoice.

If one idea should survive a skim of this whole document, it's the split in §05: a fast number that's allowed to be approximately right, and a slow number that isn't allowed to be anything else. Almost every deep dive above is either defending that split or explaining what happens at its seams.
